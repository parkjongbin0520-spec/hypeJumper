- **레퍼런스**: Celeste 플레이어 무브먼트 (코요테 타임, 점프 버퍼, 대시)
- **언어**: Python
- **프레임워크**: Pygame
- **AI 모델**: Opus 4.8 (/fast 토글)
- **기본 값 스킬**: /caveman full
- **목표**: 가볍고 구현 가능한 범위 내에서 완성도 있는 프로토타입

## 절대 규칙 (반드시 지킬 것)

1. **세션 시작 전 필요한 스킬을 탐색하고 검증 완료 후 세션을 진행한다**
2. **한 번에 하나의 파일만 수정한다**
3. **아래 항목은 반드시 승인을 받은 후 진행한다**
   - 새 파일 생성 또는 삭제
   - 파일 구조 변경
   - 다음 Phase로 넘어가는 시점
4. **일반 코드 수정/버그 수정은 자동 승인 — 단, 작업 후 WORK_LOG.md에 반드시 기록**
5. **이전 단계가 완성되지 않으면 다음 단계로 넘어가지 않는다**
6. **Phase 완료 전 반드시 QA 과정을 거친다 (아래 QA 규칙 참고)**
7. **새 시스템/클래스/메카닉 추가 전 구조 평가 보고서를 작성하고 승인을 받는다**
   
   ```
   ### 구조 평가 보고서
   - **추가할 것**: 무엇을 추가하려는지
   - **영향 범위**: 어떤 기존 코드에 영향을 주는지
   - **잠재적 버그**: 발생 가능한 문제
   - **대안**: 다른 구현 방법이 있는지
   - **결론**: 왜 이 방식을 선택했는지
   ```
8. **PLANNING.md / CLAUDE.md 업데이트 규칙 (반드시 준수)**
   - 반려 후 재작업으로 로직이 추가/변경될 경우 원본 파일을 업데이트한다
   - **추가만 허용** — 기존 내용 삭제/수정/재구성 금지
   - **추가할 위치를 WORK_LOG.md에 먼저 명시**하고 승인 후 반영
   - 업데이트 후 WORK_LOG.md에 변경 내역 기록
   - "최적화", "정리", "개선" 명목으로 기존 내용을 임의로 바꾸는 것 절대 금지

## 개발 단계 순서 (이 순서를 절대 어기지 않는다)

### Phase 1 — 플레이어 이동 완성 ✅ 완료 기준 명시

- [ ] 좌우 이동
  
  - 지상(바닥/발판): Approach 방식으로 목표 속도까지 가속, 과속 착지 시 SPEED_REDUCE로 빠르게 감속 (미끄럼 최소화)
  - 공중: 동일한 Approach 방식, AIR_MULT 배율 적용 (공중 제어력 감소)
    - vx가 PLAYER_MAX_SPEED 초과 (외적 요인): 같은 방향 입력으로 추가 가속 불가, 반대 방향 입력은 가능
    - 입력 없을 때: AIR_FRICTION으로만 서서히 감속
      
      ```
      vx=-10, 반대 입력 → -10 → -4 → +2 → +8 → +10 (유지)
      vx=-10, 입력 없음 → -10 → -8 → -6 → -4 → -2 → 0
      vx=-20, 같은 입력 → -20 → -18 → -16 → ... → -10 (한계에서 유지)
      vx=-20, 반대 입력 → -20 → -14 → -8 → -2 → +4 → +10 (유지)
      ```

- [ ] 점프 (1단 점프)
  
  - 점프 횟수: 1회 (더블 점프 없음)
  - 버튼 누르는 동안: GRAVITY_UP 적용 (약한 중력, 높이 올라감)
  - 버튼 뗀 순간부터 상승 중: GRAVITY_UP_RELEASE 적용 (강한 중력, 정점 빨리 도달)
  - 정점 이후 하강: GRAVITY_DOWN 적용 (가장 강한 중력)
  - 점프 속도(vy) 상한선 없음
    
    ```
    상승 중 버튼 유지:  vy = -10 → -8 → -6 → -4 (GRAVITY_UP)
    상승 중 버튼 뗌:    vy = -10 → -5 → 0 → +4 (GRAVITY_UP_RELEASE → GRAVITY_DOWN)
    하강:               vy = 0 → +4 → +8 → +12(MAX_FALL_SPEED)
    ```

- [ ] 코요테 타임 (절벽 끝에서 약간의 점프 유예)

- [ ] 점프 버퍼 (착지 직전 점프 입력 인식)

- [ ] **통합 버퍼 시스템** — 아래 4가지를 동일한 구조로 구현 (DASH_STRIKE_BUFFER는 Phase 3에서 추가)
  
  ```
  GROUND_COYOTE   = 지상 판정 끝난 후 N프레임 점프 허용
  WALL_COYOTE     = 벽 접촉 판정 끝난 후 N프레임 월 점프 허용
  JUMP_INPUT      = 공중 점프 입력 후 N프레임 내 착지 시 자동 점프 실행
  WALL_JUMP_INPUT = 공중 점프 입력 후 N프레임 내 벽 접촉 시 자동 월 점프 실행
  ```
  
  - 버퍼는 매 프레임 카운트다운, 0이 되면 만료
  - 조건 충족 시 자동 실행 후 즉시 버퍼 소멸
  - 각 버퍼 프레임 수는 settings.py에서 개별 튜닝 가능

- [ ] 타일맵 충돌 감지
  
  - 바닥/발판 충돌: 수직 속도 0, 지상 상태 전환
  - 벽 충돌: 수평 속도 즉시 0 → 월 슬라이드 시작
  - 천장 충돌: 충돌 직전 vy 절댓값에 비례한 밀착 프레임 부여 (밀착 중 중력 정지, 소진 후 중력 재개), 수평 가속 유지
    
    ```
    ceiling_stick_frames = min(abs(vy_before_collision) / CEILING_STICK_DIVISOR, MAX_CEILING_STICK_FRAMES)
    vy=-6  → 밀착 3프레임 → 중력 재개
    vy=-30 → 밀착 15프레임 → 중력 재개
    ```
  - 낙하 최대 속도: 중력에 의한 vy는 MAX_FALL_SPEED를 초과할 수 없음
  - **상한선 규칙 (반드시 준수)**
    - 상한선 있음: 낙하 속도(중력), 천장 밀착 프레임
    - 상한선 없음: 점프 속도(위 방향 vy), 좌우 가속 (대시/발판 관성/하이퍼 대시 등 외부 힘 전달)

- [ ] **플레이어 상태머신** — PlayerState enum을 Phase 1에 미리 전체 정의, Phase 1에서는 NORMAL 로직만 구현
  
  - enum 전체 미리 정의: NORMAL / DASH / DASH_STRIKE / GRAB_SEEKING / GRAB_READY / GRAB_ACTIVE
  - Phase 1: NORMAL 로직만 구현 (중력/월슬라이드/점프/웅크리기 조건 분기)
  - Phase 2: DASH 로직 추가, Phase 3: GRAB 로직 추가 (enum은 이미 있으므로 로직만 추가)
  - 상태머신을 if/else로 직접 구현 금지, PlayerState enum 사용

- [ ] **웅크리기 (Duck)** — 지상 + 아래 입력
  
  - 히트박스 높이 축소 (DUCK_HITBOX_HEIGHT)
  - 이동 입력 시 웅크리기 해제 → 일반 이동
  - 웅크리기 중 점프 → 웅크리기 해제 → 일반 점프
  - 웅크리기 중 대시 → 대각선 아래 대시 (하이퍼 트리거 가능)

- [ ] **패스트 폴 (Fast Fall)** — 공중에서 아래 방향키 입력 시 낙하 가속
  
  - 공중 하강 중 + 아래 입력: GRAVITY_DOWN 대신 FAST_FALL_GRAVITY 적용, 최대 속도 FAST_MAX_FALL
  - 월 슬라이드 중 + 아래 입력: 월 슬라이드 중력 감소 무시, 빠르게 내려감

- [ ] **월 슬라이드** — 공중에서 벽에 붙었을 때 WALL_SLIDE_GRAVITY로 천천히 내려오기

- [ ] **월 점프** — 벽에 붙은 상태에서 점프 시 벽 반대 방향으로 튕기기

- [ ] **월 점프 버퍼** — 벽 접촉 판정 유예 (벽에서 떨어진 직후에도 월 점프 인식)

- [ ] **움직이는 발판 탑승** — 발판 위에 서있을 때 발판의 이동 벡터를 플레이어 위치에 그대로 적용 (플레이어 입력 속도와 독립적인 별도 레이어로 합산)

- [ ] **발판 관성 전달** — 발판에서 점프 시 발판의 x/y 속도 벡터를 플레이어에게 전달
  
  - 위로 올라가는 발판: 점프 높이 증가
  - 아래로 내려가는 발판: 점프 높이 감소
  - 수평 이동 발판: 점프 후에도 수평 관성 유지
  - **관성 감소 방식**: 공기 저항 방식 — 점프 후 매 프레임 관성에 감쇠 계수(AIR_FRICTION)를 곱해서 서서히 0으로 수렴 (즉시 소멸 또는 완전 유지 방식 사용 금지)

- [ ] **발판 끼임 사망 (Crush Death)** — 움직이는 발판에 의해 플레이어가 다른 솔리드 사이에 핀되면 사망(리스폰)
  
  - 모델: 셀레스트식 밀기+핀 (옵션 B). 판정은 발판 이동 직후·플레이어 자체 move 이전 (속도 비의존)
  - 캐리(탑승 중) 케이스: 발판 델타로 함께 이동 후 다른 솔리드와 겹치면 사망 (벽/천장으로 캐리)
  - 밀기(비탑승) 케이스: 발판 진행 방향(dx/dy 4부호)으로 플레이어를 밀어낸 뒤, 반대편 솔리드에 막혀 여전히 겹치면 사망
  - 밀어낼 공간이 있으면 회피(생존) — 좌/우/위/아래 4방향 동일 코드로 처리
  - 상세 로직은 PLANNING.md "발판 끼임 사망 (Crush Death)" 참고

## 핵심 아키텍처 규칙 (반드시 이 구조로 구현)

### 클래스 상속 구조

```
Entity → Actor  → Player, Enemy, NTT
       → Solid  → MovingPlatform
       → Trigger → JumpPad, Spring, Hazard
       → Projectile (직접 상속, Actor 아님)
```

- Actor: 이동/중력/충돌 공통 로직
- Solid: 막히는 오브젝트, Actor 이동 차단
- Trigger: target_layers로 반응 대상 필터링, 스스로 충돌 체크
- Projectile: 중력 없음, Solid 통과 없음, WALL 충돌 시 소멸

### 플레이어 속도 구조

수평 속도는 반드시 두 컴포넌트로 분리해서 관리한다:

```python
vx_input    # 플레이어 입력에 의한 속도 (-PLAYER_MAX_SPEED ~ +PLAYER_MAX_SPEED)
vx_external # 외적 요인에 의한 속도 (대시, 발판 관성 등, 상한선 없음)
# 최종 속도: vx = vx_input + vx_external
```

- `vx_input`: 지상에서만 변경, 공중에서 방향 전환 가능하나 PLAYER_MAX_SPEED 한계
- `vx_external`: AIR_FRICTION으로 매 프레임 감쇠, 외적 속도 초과 시 같은 방향 입력으로 추가 가속 불가
- 두 컴포넌트를 하나의 변수로 합치는 것 금지

### 통합 버퍼 시스템

4가지 버퍼를 동일한 클래스 구조로 구현한다 (Phase 3에서 DASH_STRIKE_BUFFER 추가):

```python
class InputBuffer:
    def __init__(self, max_frames):
        self.frames = 0
        self.max_frames = max_frames
    def set(self):       # 버퍼 활성화
    def tick(self):      # 매 프레임 카운트다운
    def is_active(self): # 활성 여부 확인
    def consume(self):   # 조건 충족 시 소멸
# Phase 1: GROUND_COYOTE, WALL_COYOTE, JUMP_INPUT, WALL_JUMP_INPUT
# Phase 3 추가: DASH_STRIKE_BUFFER
```

### 상태머신

PlayerState enum과 별도 클래스로 분리 구현, 물리 로직과 혼합 금지:

```python
class PlayerState(Enum):
```
